# О проекте

**AsyncEngine** — это внешняя компонента Native API для платформы **1С:Предприятие**, предназначенная для запуска долговременно работающих асинхронных Python-приложений непосредственно внутри процесса 1С.

В отличие от традиционного взаимодействия с Python через запуск внешнего процесса (`python.exe`) или HTTP-сервисов, AsyncEngine встраивает полноценный интерпретатор Python непосредственно в адресное пространство процесса 1С.

Это позволяет:

- исключить запуск внешних процессов;
- избавиться от постоянного обмена через HTTP;
- минимизировать задержки передачи данных;
- выполнять десятки и сотни асинхронных задач одновременно;
- использовать любую современную Python-экосистему непосредственно из 1С.

---

AsyncEngine не является простой оберткой над Embedded Python.

Проект представляет собой полноценную платформу исполнения (Runtime), состоящую из нескольких независимых уровней.

```text
                    1С:Предприятие

                           │
                           │ Native API
                           ▼

                    AsyncEngine.dll
                     (C++ Runtime)

                           │

         Embedded Python + pybind11 + GIL

                           │

                     async_core.py

                           │

                asyncio Event Loop

                           │

          ┌────────────────────────────────┐
          │                                │
          ▼                                ▼

    Plugin Instance                  Plugin Instance

          │                                │

     asyncio.Queue                    asyncio.Queue

          │                                │

     callback()                       callback()

          └──────────────┬─────────────────┘
                         ▼

                 ExternalEvent()

                         │

                         ▼

                    Код 1С
```

Таким образом C++ отвечает исключительно за интеграцию платформы 1С и Python, а вся прикладная логика полностью переносится в Python-плагины.

Подобное разделение обязанностей делает систему расширяемой и позволяет создавать новые возможности без изменения C++-компоненты.

# Для чего создавался проект

Основная идея проекта состоит в том, чтобы сделать Python полноценным асинхронным исполнительным окружением внутри платформы 1С.

В классической интеграции с Python обычно используется один из следующих подходов:

- запуск python.exe;
- COM;
- HTTP REST;
- gRPC;
- файловый обмен;
- RabbitMQ;
- очереди сообщений.

Каждый из этих вариантов требует запуска отдельного процесса.

Следствием являются:

- большие задержки;
- сложная установка;
- проблемы с обменом данными;
- необходимость постоянного сопровождения внешнего сервиса.

AsyncEngine решает эту проблему иначе.

Python становится частью процесса 1С.

Это позволяет работать практически без накладных расходов на межпроцессное взаимодействие.

# Основные возможности

На текущий момент проект поддерживает следующие возможности.

## Embedded Python

Компонента самостоятельно загружает библиотеку Python во время работы.

Версия Python не зашивается в код.

Во время инициализации пользователь передает:

- путь к Python;
- путь к модулям;
- основной Python-модуль;
- используемую версию Python.

После этого библиотека автоматически загружается посредством `LoadLibrary()`.

Это позволяет использовать различные версии Python без перекомпиляции компоненты.

---

## Полностью асинхронная модель

Вся работа Python осуществляется внутри одного Event Loop.

Каждый плагин представляет собой отдельную coroutine.

Например одновременно могут работать:

- Socket.IO сервер;
- Socket.IO клиент;
- Web Scraper;
- Telegram Bot;
- MQTT клиент;
- AI Agent;
- OCR обработчик;
- HTTP API.

Все перечисленные задачи используют один Event Loop и не создают дополнительных потоков Python.

---

## Изоляция экземпляров

Каждый экземпляр плагина получает:

- собственный TaskID;
- собственную очередь сообщений;
- собственную coroutine;
- собственный жизненный цикл.

Поэтому одновременно можно запускать несколько экземпляров одного и того же плагина.

Например:

```text
socketio_client
    TaskID = Client1

socketio_client
    TaskID = Client2

socketio_client
    TaskID = Client3
```

Каждый экземпляр полностью независим.

---

## Двусторонний обмен сообщениями

Архитектура проекта поддерживает двусторонний обмен данными.

Из 1С в Python:

```text
RunPlugin()

↓

Plugin

↓

asyncio.Queue
```

Из Python обратно в 1С:

```text
callback()

↓

AsyncEngine

↓

ExternalEvent()

↓

1С
```

Это позволяет создавать реактивные приложения без постоянного опроса состояния со стороны 1С.

---

## Динамическая загрузка плагинов

При запуске используется динамический импорт Python-модулей.

Плагин выбирается строковым именем.

Например:

```1C
RunPlugin("socketio_server", ...)
```

или

```1C
RunPlugin("web_scrapper", ...)
```

или

```1C
RunPlugin("my_plugin", ...)
```

Компонента не знает заранее о существовании конкретного плагина.

Весь механизм основан на динамическом импорте Python.

Это означает, что добавление нового плагина не требует изменения C++ кода.

Достаточно разместить новый модуль в каталоге `plugins`.

# Почему AsyncEngine

Главная особенность проекта заключается не в использовании Embedded Python.

Главная особенность — разделение ответственности между уровнями системы.

| Уровень | Ответственность |
|----------|-----------------|
| 1С | пользовательский интерфейс, бизнес-логика, документы |
| C++ | интеграция платформы и Python |
| async_core.py | диспетчеризация и управление жизненным циклом |
| Plugin | прикладная логика |
| asyncio | выполнение асинхронных операций |

Благодаря этому C++ практически не содержит бизнес-логики.

Его задача — предоставить безопасную и производительную среду исполнения.

Все остальное реализуется обычными Python-плагинами.

Именно поэтому проект можно рассматривать как Runtime для выполнения асинхронных расширений платформы 1С.

# Жизненный цикл

Работа Runtime состоит из пяти этапов:

```text
InitPython()

        │

        ▼

Embedded Python

        │

        ▼

Event Loop

        │

        ▼

RunPlugin()

        │

        ▼

Работа плагинов

        │

        ▼

Done()
```

После инициализации создается Embedded Python и запускается отдельный поток с `asyncio Event Loop`.

При вызове `RunPlugin()` Runtime динамически загружает модуль, создает экземпляр плагина и выделяет ему собственную очередь сообщений (`asyncio.Queue`).

Во время работы обмен данными происходит в двух направлениях:

- **1С → Python** — через `SendMessageToPlugin()`;
- **Python → 1С** — через `callback()`, который преобразуется в `ExternalEvent()`.

При завершении работы компоненты Runtime останавливает Event Loop, завершает все задачи и корректно освобождает ресурсы Python.
